# Phase 3 Data Pipeline

This directory contains the complete data pipeline for the **task-aware LLM orchestration** project.
It loads tasks from four benchmark datasets, runs a 500-episode simulation logging results in strict JSONL format, computes retroactive difficulty labels, and generates synthetic perturbations for harder task variants.

---

## Directory Structure

```
phase3/data/
├── loaders/
│   ├── gsm8k_loader.py        # GSM8K (reasoning)
│   ├── humaneval_loader.py    # OpenAI HumanEval (code)
│   ├── triviaqa_loader.py     # TriviaQA RC (qa)
│   └── arc_loader.py          # ARC-Challenge (reasoning)
├── pipeline.py                # Unified task sampler (seed=42)
├── logger.py                  # Schema-validated JSONL logger
├── run_pipeline.py            # 500-episode run script
├── label_difficulty.py        # Retroactive difficulty labeler
├── generate_perturbations.py  # Synthetic perturbation generator
├── runs/
│   └── phase3_run_001.jsonl   # Generated by run_pipeline.py
├── task_difficulty_labels.jsonl  # Generated by label_difficulty.py
└── perturbations.jsonl           # Generated by generate_perturbations.py
```

## Dataset Downloads

HuggingFace datasets are downloaded automatically on first use. To pre-download
them manually:

```bash
# GSM8K (reasoning — grade-school math)
python -c "from datasets import load_dataset; load_dataset('gsm8k', 'main')"

# OpenAI HumanEval (code generation)
python -c "from datasets import load_dataset; load_dataset('openai_humaneval')"

# TriviaQA RC (question answering)
python -c "from datasets import load_dataset; load_dataset('trivia_qa', 'rc')"

# ARC-Challenge (reasoning — science questions)
python -c "from datasets import load_dataset; load_dataset('ai2_arc', 'ARC-Challenge')"
```

Datasets are cached locally under `~/.cache/huggingface/datasets/` by default.

## Reproducing the Full Pipeline

> **Note:** All scripts use a fixed random seed of **`seed=42`** to guarantee
> deterministic, reproducible outputs. Running any script twice with the same
> seed will always produce identical results.

### Step 1a — Run the 500-episode simulation

```bash
# From the project root (task-aware-llm-orchestration/)
python phase3/data/run_pipeline.py
```

Output: `phase3/data/runs/phase3_run_001.jsonl` (1500 lines — 500 tasks × 3 workflows, not committed)

### Step 1b — Run the bandit training log

```bash
python phase3/data/run_bandit_training.py
```

Output: `phase3/data/runs/phase3_bandit_train.jsonl` (500 lines — one routing decision per task, not committed)

This is the training log the Phase 4 LinUCB policy learns from. It makes one
random routing decision per task rather than exhaustively evaluating all three
workflows.

### Step 2 — Compute retroactive difficulty labels

```bash
python phase3/data/label_difficulty.py
```

Output: `phase3/data/task_difficulty_labels.jsonl`

Labels are assigned per `task_id` based on the quality-score gap between
workflow W1 and W3 runs:

| Gap (W3 − W1) | Label           |
|---------------|-----------------|
| > 0.3         | high difficulty |
| 0.1 – 0.3     | medium          |
| < 0.1         | easy            |

### Step 3 — Generate synthetic perturbations

```bash
python phase3/data/generate_perturbations.py
```

Output: `phase3/data/perturbations.jsonl` (300 records: 100 tasks × 3 perturbation types)

Perturbation types applied to each task:

| Type               | Description                                      |
|--------------------|--------------------------------------------------|
| `irrelevant_info`  | Prepends an unrelated distractor sentence        |
| `negation`         | Negates the first matching verb phrase           |
| `ambiguous_pronoun`| Appends a sentence with an ambiguous pronoun     |

## JSONL Schema

Each episode record written by `run_pipeline.py` has the following schema:

| Field            | Type           | Description                              |
|------------------|----------------|------------------------------------------|
| `task_id`        | string         | Unique task identifier                   |
| `task_text`      | string         | The task prompt / question               |
| `task_class`     | string         | `qa`, `reasoning`, `code`, `explanation` |
| `feature_vector` | list[float]    | Numeric feature representation           |
| `workflow_id`    | string         | `W1`, `W2`, or `W3`                      |
| `output_text`    | string         | Model response (mock in this run)        |
| `quality_score`  | float          | Quality of the response [0.0, 1.0]       |
| `cost_tokens`    | integer        | Number of tokens consumed                |
| `latency_ms`     | integer        | Response latency in milliseconds         |
| `reward`         | float          | `quality_score − λ × normalised_cost`   |
| `ground_truth`   | string or null | Reference answer (null for HumanEval)    |
| `episode_id`     | integer        | Sequential episode index                 |
| `lambda_value`   | float          | Cost-quality trade-off weight (λ)        |

The `logger.py` module enforces this schema on every write and immediately
raises a `ValueError` if any field is missing or has the wrong type.

## Reproducibility

All randomness is controlled by **`seed=42`**. This applies to:

- Task sampling order in `pipeline.py`
- Mock feature vector generation in `run_pipeline.py`
- Task sampling in `generate_perturbations.py`

Changing the seed will produce a different but still fully reproducible ordering.
